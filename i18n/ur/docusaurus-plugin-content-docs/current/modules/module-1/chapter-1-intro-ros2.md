# چیپٹر 1: ROS 2 کا تعارف

## جائزہ

یہ چیپٹر ROS 2 (روبوٹ آپریٹنگ سسٹم 2) کے بنیادی تصورات کا تعارف دیتا ہے، روبوٹکس کے لیے نسل کے بعد والے مڈل ویئر فریم ورک۔ ہم ROS 1 سے ROS 2 تک کی ترقی کو جائزہ دیں گے، بنیادی معماری کی تبدیلیوں کو سمجھیں گے، اور یہ دیکھیں گے کہ ROS 2 جدید روبوٹکس ایپلی کیشنز بشمول ہیومنوائڈ روبوٹس کی ضروریات کو کیسے پورا کرتا ہے۔ اس چیپٹر کے اختتام تک، آپ کو ROS 2 کی معماری کا مضبوط تاثر ہوگا اور آپ اپنی پہلی ROS 2 ایپلی کیشنز تیار کرنے کے لیے تیار ہوں گے۔

ROS 2 ROS 1 پر ایک قابل ذکر ترقی کی نمائندگی کرتا ہے، جس میں جدید سافٹ ویئر انجینئرنگ کے طریقے، بہتر سیکورٹی، حقیقی وقت کی صلاحیتیں، اور تجارتی اور صنعتی ایپلی کیشنز کی حمایت شامل ہے۔ یہ فریم ورک پیچیدہ روبوٹک سسٹم کی تیز ترقی کے قابل بنانے والے اوزاروں اور لائبریریز کا امیر سیٹ فراہم کرتا ہے۔

## سیکھنے کے اہداف

- ROS ورژن 1 سے 2 تک کی تاریخ اور ترقی کو سمجھنا
- ROS 1 اور ROS 2 کے درمیان کلیدی معماری کے فروق کی وضاحت کرنا
- ROS 2 کے بنیادی اجزاء اور تصورات کی شناخت کرنا
- بنیادی ROS 2 ترقی کا ماحول قائم کرنا
- اپنے پہلے ROS 2 نوڈس چلانا اور بنیادی مواصلت کو سمجھنا
- سسٹم کے اندر کی نگرانی کے لیے ROS 2 کمانڈ لائن ٹولز کا استعمال کرنا

## کلیدی تصورات

### ROS 2 بمقابلہ ROS 1

ROS 2 ROS 1 کی کئی حدود کو حل کرتا ہے بشمول واحد تھریڈڈ ایکزیکیوشن ماڈل، سیکورٹی کی خصوصیات کی کمی، اور حقیقی وقت کی حمایت کا فقدان۔ نئی معماری DDS (ڈیٹا ڈسٹری بیوشن سروس) کو بنیادی مواصلت کے مڈل ویئر کے طور پر استعمال کرتی ہے، جو قابل اعتماد پیغام کی ترسیل اور دریافت کے میکانزم فراہم کرتا ہے۔

### DDS مڈل ویئر

DDS (ڈیٹا ڈسٹری بیوشن سروس) حقیقی وقت، قابل توسیع، اور خرابی برداشت ڈیٹا ایکسچینج کے لیے ایک OMG معیار ہے۔ ROS 2 میں، DDS نوڈ کی دریافت، پیغام کی راؤٹنگ، اور سروس کی معیار (QoS) پالیسیز کو سنبھالتا ہے۔ متعدد DDS ایمپلیمنٹیشن کی حمایت کی جاتی ہے بشمول Fast DDS، Cyclone DDS، اور RTI Connext DDS۔

### نوڈس اور مواصلت

ROS 2 میں نوڈس وہ عمل ہیں جو ایک حساب کتاب انجام دیتے ہیں۔ وہ ٹاپکس (شائع کرنا-سبسکرائب کرنا)، سروسز (درخواست-جواب)، اور ایکشنز (ہدف کے مطابق مواصلت) کے ذریعے بات چیت کرتے ہیں۔ ہر نوڈ کا ایک منفرد نام ہوتا ہے اور وہ ایک وقت میں متعدد مواصلت کے نمونوں میں حصہ لے سکتا ہے۔

### سروس کی معیار (QoS)

QoS ترتیبات مواصلت کے رویے کو باریکی سے ایڈجسٹ کرنے کی اجازت دیتی ہیں بشمول قابل اعتمادی، م durable، اور تاریخ کی پالیسیز۔ یہ ترتیبات حقیقی وقت اور حفاظتی اہمیت والی روبوٹک ایپلی کیشنز کے لیے انتہائی اہم ہیں، جو ڈیولپرز کو پیغام کی ترسیل اور ڈیٹا کی مسلسل دستیابی کے لیے ضروریات کی وضاحت کرنے کی اجازت دیتی ہیں۔

## تکنیکی گہرائی

### معماری کی ترقی

ROS 2 اصل ROS فریم ورک کا مکمل دوبارہ ڈیزائن ہے تاکہ حقیقی وقت کی کارکردگی، سیکورٹی، اور قابل توسیع ہونے کی ضروریات کو پورا کیا جا سکے۔ کلیدی معماری کی تبدیلیاں شامل ہیں:

**DDS انضمام**: ROS 2 DDS کو بنیادی مواصلت کے مڈل ویئر کے طور پر استعمال کرتا ہے، جو حقیقی وقت، قابل توسیع، اور خرابی برداشت ڈیٹا ایکسچینج کے لیے ایک معیاری انٹرفیس فراہم کرتا ہے۔ یہ ROS 2 کو حقیقی وقت کی ایپلی کیشنز کی حمایت کرنے کی اجازت دیتا ہے اور بہتر نیٹ ورک دریافت اور پیغام کی ترسیل کی ضمانت فراہم کرتا ہے۔

**میلٹی تھریڈڈ ایکزیکیوشن**: ROS 1 کے واحد تھریڈڈ ایکزیکیوشن ماڈل کے برعکس، ROS 2 میلٹی تھریڈڈ ایکزیکیوشن کی حمایت کرتا ہے، جو پیچیدہ روبوٹک ایپلی کیشنز کے لیے بہتر کارکردگی کی اجازت دیتا ہے۔

**سیکورٹی**: ROS 2 میں سیکورٹی کی خصوصیات شامل ہیں بشمول تصدیق، رسائی کنٹرول، اور انکرپشن، جو اسے تجارتی اور صنعتی ایپلی کیشنز کے لیے مناسب بناتا ہے۔

### بنیادی اجزاء

**نوڈس**: ROS 2 ایپلی کیشنز کے بنیادی عناصر۔ ہر نوڈ آزادانہ طور پر چلتا ہے اور دوسرے نوڈس کے ساتھ پیغامات کے ذریعے بات چیت کرتا ہے۔

**ٹاپکس**: نامزد بسز جن کے ذریعے نوڈس ایک شائع کرنا-سبسکرائب کرنا نمونہ کے ساتھ پیغامات کا تبادلہ کرتے ہیں۔ شائع کنندہ ٹاپکس پر پیغامات بھیجتے ہیں، اور سبسکرائبرز ٹاپکس سے پیغامات وصول کرتے ہیں۔

**سروسز**: درخواست-جواب مواصلت کا نمونہ جہاں ایک کلائنٹ ایک درخواست بھیجتا ہے اور ایک سرور سے جواب کا انتظار کرتا ہے۔

**ایکشنز**: ہدف کے مطابق مواصلت جس میں فیڈ بیک اور نتیجہ رپورٹنگ شامل ہے، جو طویل چلنے والے آپریشنز کے لیے مناسب ہے۔

### سروس کی معیار کی پالیسیز

ROS 2 QoS پالیسیز مواصلت کے رویے پر باریکی سے کنٹرول فراہم کرتی ہیں:

- **قابل اعتمادی پالیسی**: یہ کنٹرول کرتی ہے کہ پیغامات کو قابل اعتماد طریقے سے یا بہترین کوشش کے ساتھ ترسیل کیا جائے
- **م durable پالیسی**: یہ تعین کرتی ہے کہ کیا پیغامات دیر سے شامل ہونے والے سبسکرائبرز کے لیے مسلسل دستیاب رہیں گے
- **تاریخ پالیسی**: یہ کنٹرول کرتی ہے کہ ہر ٹاپک کے لیے کتنے پیغامات محفوظ کیے جاتے ہیں
- **ڈیڈ لائن پالیسی**: پیغام کی ترسیل کے لیے وقت کی پابندیوں کی وضاحت کرتی ہے
- **لائیو لینس پالیسی**: نوڈ دستیابی کی نگرانی کرتی ہے

### ROS 2 ایکو سسٹم

ROS 2 ایکو سسٹم میں شامل ہے:

- **کلائنٹ لائبریریز**: rclpy (پائی تھون)، rclcpp (سی++)، اور دیگر
- **اوزار**: اندر کی نگرانی، ڈیبگنگ، اور وژولائزیشن کے لیے کمانڈ لائن ٹولز
- **پیکیج سسٹم**: colcon بلڈ سسٹم اور پیکیج مینجمنٹ
- **لاؤنچ سسٹم**: متعدد نوڈس کو ترتیبات کے ساتھ شروع کرنے کے لیے لاؤنچ فائلز

## کوڈ کی مثالیں

### بنیادی شائع کنندہ نوڈ

```python
import rclpy
from rclpy.node import Node
from std_msgs.msg import String

class MinimalPublisher(Node):

    def __init__(self):
        super().__init__('minimal_publisher')
        self.publisher_ = self.create_publisher(String, 'topic', 10)
        timer_period = 0.5  # سیکنڈ
        self.timer = self.create_timer(timer_period, self.timer_callback)
        self.i = 0

    def timer_callback(self):
        msg = String()
        msg.data = 'ہیلو دنیا: %d' % self.i
        self.publisher_.publish(msg)
        self.get_logger().info('شائع کر رہا ہے: "%s"' % msg.data)
        self.i += 1

def main(args=None):
    rclpy.init(args=args)
    minimal_publisher = MinimalPublisher()
    rclpy.spin(minimal_publisher)
    minimal_publisher.destroy_node()
    rclpy.shutdown()

if __name__ == '__main__':
    main()
```

### بنیادی سبسکرائبر نوڈ

```python
import rclpy
from rclpy.node import Node
from std_msgs.msg import String

class MinimalSubscriber(Node):

    def __init__(self):
        super().__init__('minimal_subscriber')
        self.subscription = self.create_subscription(
            String,
            'topic',
            self.listener_callback,
            10)
        self.subscription  # غیر استعمال شدہ متغیر کی وارننگ کو روکنے کے لیے

    def listener_callback(self, msg):
        self.get_logger().info('میں نے سنا: "%s"' % msg.data)

def main(args=None):
    rclpy.init(args=args)
    minimal_subscriber = MinimalSubscriber()
    rclpy.spin(minimal_subscriber)
    minimal_subscriber.destroy_node()
    rclpy.shutdown()

if __name__ == '__main__':
    main()
```

### نوڈس چلانا

ان نوڈس کو چلانے کے لیے، انہیں الگ الگ پائی تھون فائلز کے طور پر محفوظ کریں اور انہیں مختلف ٹرمنلز میں چلائیں:

ٹرمنل 1:

```bash
ros2 run my_package minimal_publisher
```

ٹرمنل 2:

```bash
ros2 run my_package minimal_subscriber
```

## عام مسائل

- **نوڈ دریافت کے مسائل**: نیٹ ورک کی ترتیب یا DDS ایمپلیمنٹیشن کے تنازعات کی وجہ سے نوڈس بات چیت کرنے میں ناکام ہو سکتے ہیں۔ یقینی بنائیں کہ دونوں نوڈس ایک ہی DDS ایمپلیمنٹیشن کا استعمال کر رہے ہیں اور ایک ہی نیٹ ورک پر ہیں۔
- **QoS عدم مطابقت**: غیر مطابقت پذیر QoS ترتیبات والے شائع کنندہ اور سبسکرائبرز بات چیت نہیں کریں گے۔ ہمیشہ قابل اعتماد مواصلت کے لیے مطابقت پذیر QoS پالیسیز کو یقینی بنائیں۔
- **ریسورس مینجمنٹ**: نوڈس کو مناسب طریقے سے تباہ کرنے کو بھولنا میموری لیکس کا سبب بن سکتا ہے۔ ہمیشہ بند کرنے سے پہلے `destroy_node()` کال کریں۔
- **پیکیج انحصار**: غائب یا غلط پیکیج انحصار نوڈس کو چلنے سے روک سکتا ہے۔ یقینی بنائیں کہ تمام ضروری پیکیجز `package.xml` میں مناسب طریقے سے بیان کیے گئے ہیں۔
- **ROS ڈومین ID**: ایک ہی نیٹ ورک پر متعدد ROS 2 سسٹم کو مداخلت سے بچنے کے لیے مختلف ڈومین ID کی ضرورت ہوتی ہے۔

## چیک پوائنٹس / مینی ایکسائزز

1. ایک ROS 2 ترقی کا ماحول قائم کریں اور ٹاکر/لیسنر ڈیمو چلا کر یہ تصدیق کریں کہ یہ کام کرتا ہے
2. ایک شائع کنندہ نوڈ تیار کریں جو متعدد فیلڈز والے ایک حسب ضرورت پیغام کی قسم کو شائع کرتا ہے
3. چلتے ہوئے سسٹم کی نگرانی کے لیے ROS 2 کمانڈ لائن ٹولز کا استعمال کریں: `ros2 node list`, `ros2 topic list`, `ros2 topic echo`
4. شائع کنندہ کو مختلف QoS ترتیبات استعمال کرنے کے لیے تبدیل کریں اور اثرات دیکھیں
5. ایک لاؤنچ فائل تیار کریں جو دونوں شائع کنندہ اور سبسکرائبر نوڈس کو ایک ہی وقت میں شروع کرے
